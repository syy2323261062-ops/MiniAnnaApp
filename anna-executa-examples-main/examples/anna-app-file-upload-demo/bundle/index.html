anna-app Host Upload Demo

Demonstrates the host/uploadFile reverse-RPC — inline, negotiate and confirm — persisting a binary blob to short-lived host storage (~30 min TTL, not enumerable). See the host docs.

  • Bytes originate in the Executa, not the browser. A browser-picked File can only reach a subprocess over the JSON-RPC stdio channel, which caps the size and defeats negotiate. So the bundled Executa generates the payload locally (make_sample) and uploads it by path (host_upload_path) — the bytes go plugin → R2 directly.
  • inline — payloads ≤ 8 MiB ride one reverse-RPC round-trip as base64.
  • negotiate + confirm — larger payloads: the host signs an R2 PUT URL, the Executa streams the file straight to R2, then confirm registers it. This is the large-file path.

Run with anna-app dev while signed in (mints an app_session_token) — Host Upload does not require --storage aps. The host enforces the user's upload_grant: a MIME whitelist (the dev grant allows image/*, application/pdf, text/plain, application/json — samples upload as application/pdf) and a per-file size cap (25 MiB in dev). Over the cap → UPLOAD_TOO_LARGE (-32204); a non-whitelisted MIME → UPLOAD_MIME_REJECTED (-32210) — both before any R2 round-trip.

1. Preset uploads

Each button asks the Executa to make_sample a payload of that size, then host_upload_path it. Watch the chosen mode flip from inline to negotiate+confirm across the 8 MiB cap — and the last one trip the per-file quota (dev cap 25 MiB).

    2. Upload a local file by path

    Hand the Executa an absolute path it can read on this machine. It picks inline or negotiate+confirm by size.
